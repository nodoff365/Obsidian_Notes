1. Onpremise Storage

	1.1 DAS(Direct Attache Storage): 물리적인 전송 매체(IDE, SATA, SCASI)를 이용해서 Storage를 직접적으로 연결
		1.1.1 장점 - 속도가 빠름, 비용 저렴, 안정적
		1.1.2 단점 - 원거리 구성 불가, 확장 제한적

	1.2 NAS(Network Area Storage): 기존 네트워크 환경(100Mbps, 1Gbps)을 이용해서 Storage를 연결
		1.2.1 장점 - 구성 용이, 비용 저렴, 원거리 구성 가능
		1.2.2 단점 - 속도 느림, 네트워크에 병목 현상이 발생하면 더 느려짐

	1.3 SAN(Storage Area Network): Storage 영역에 별도의 Network 구성, Fiber Channel
		1.3.1 장점 - 속도가 빠름(16Gbps), 안정적, 원거리 구성 가능
		1.3.2 단점 - 고가의 비용, 구성 쉽지 않음(별도의 SAN Switch, HBA Card)

2. Cloud Storage
	2.1 Block Storage: HDD 형태의 가상 Storage. 실제 Onpremise 환경의 Block Storage와 동일하게 동작

	2.2 Object Storage: Rest API(객체별로 URL을 제공), 거의 무제한의 용량 제공, 접근제어 가능, 버전관리 가능
	AWS(S3), Azure(Blob)

1. Disk 사용절차
	1.1 Linux: 파티션 -> 파일시스템 -> Mount

	1.2 Windows: 파티션 -> 파일시스템 -> 드라이브명 할당

	1.3 파티션
		1.3.1 물리적인 디스크를 논리적인 단위로 분리
		1.3.2 Disk를 사용하기 위해서는 반드시 1개 이상의 논리적인 파티션이 필요
		1.3.3 MBR 방식에서 주 파티션을 4개까지만 생성 가능
			1.3.3.1 주 파티션중 오로지 1개의 파티션을 확장파티션으로 생성 후 논리 파티션을 12개까지 생성가능
		1.3.4 UEFI 방식은 128개의 주파티션 생성가능

	1.4 파일시스템: 파일 및 디렉토리를 효과적으로 관리하는 시스템
		1.4.1 Windows: FAT16, FAT32, NTFS
		1.4.2 Linux: EXT2, EXT3, EXT4, XFS
			1.4.2.1 EXT3, EXT4: Journaling file system 파일이나 디렉토리를 생성, 삭제 시 Log를 기록
						 이 Log를 통해서 파일 복구 가능, 단 속도가 느려짐
						 EXT3의 느린 속도를 개선한 버전이 EXT4

	1.5 Mount : Linux System에서는 물리적인 장치 사용하기 위해서는 반드시 디렉토리와 연결하는 작업이 필요

### 1. 디스크 확인

```bash
lsblk

# 예시
NAME        MAJ:MIN RM  SIZE RO TYPE MOUNTPOINTS
sda           8:0    0   50G  0 disk 
├─sda1        8:1    0    1G  0 part /boot
└─sda2        8:2    0   49G  0 part 
  ├─rl-root 253:0    0   47G  0 lvm  /
  └─rl-swap 253:1    0    2G  0 lvm  [SWAP]
sdb           8:16   0   10G  0 disk
```

현재는 `sdb`만 있고 `sdb1`은 없음

### 4. 파일시스템 생성

```bash
mkfs -t ext3 /dev/sdb1
mkfs -t ext4 /dev/sdb2
mkfs.xfs /dev/sdb3
```

mount /dev/sdb3 /f3
