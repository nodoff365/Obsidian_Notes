**도입 (07 검증 진입 전)** 
앞에서 설계하고 구축한 인프라가 실제로 의도대로 동작하는지 8개 영역으로 나누어 진행한 결과를 보여드리겠습니다. 첫번째 영역은 보안 영역입니다. 

**슬라이드 1 — 보안 ① VMSS 공인 IP 미할당 · Firewall 강제 경유**
제로 트러스트 설계의 출발점은 웹 서버를 외부에 직접 노출하지 않는 것입니다. 웹 서버가 공인 IP를 가지면 공격자가 Application Gateway를 우회해 직접 접근할 수 있기 때문에, VMSS(버추얼 머신 스케일 셋츠)는 공인 IP 없이 게이트웨이 뒤에 두었습니다. 실제로 VMSS 인스턴스에는 공인 IP가 할당되어 있지 않은 모습이고, VMSS 안에서 시스템의 공용 IP 주소를 확인하는 명령어(`curl ifconfig.me`)를 실행했을 때 인스턴스 IP가 아니라 Firewall 공인 IP가 반환된 모습입니다. 이를 통해 외부 직접 접근이 차단되고, 모든 아웃바운드가 Firewall을 강제 경유함이 검증되었습니다.

**슬라이드 2 — 보안 ① UDR 라우트 테이블 (Central)**
Firewall 강제 경유는 저절로 되는 것이 아니라 사용자 지정 라우팅(UDR)으로 강제해야 합니다. 기본 상태로 두면 트래픽이 인터넷으로 바로 나가 Firewall을 우회하기 때문입니다. Web 서브넷과 Private Endpoint 서브넷에 라우트 테이블을 적용하여, 모든 목적지 경로(0.0.0.0/0)가 Firewall로 향하도록 지정되어 있는 모습입니다. 이로써 앞 슬라이드의 강제 경유가 이 라우팅 때문임이 검증되었습니다.

**슬라이드 3 — 보안 ① UDR 라우트 테이블 (Japan)**
재해복구 리전이 주 리전과 보안 수준이 다르면 페일오버(장애가 발생했을 때 예비 시스템으로 자동 전환하여 중단 없이 서비스를 계속 제공하는 기술) 순간 보안 구멍이 생기므로, 동일한 라우팅을 적용했습니다. 양 리전이 대칭으로 구성되어, 어느 리전으로 전환(절체) 되더라도 동일한 보안 정책이 유지됨이 검증되었습니다.

**슬라이드 4 — 보안 ① WAF 정상/차단 응답**
네트워크 계층 차단만으로는 SQL 인젝션(DB를 조작하는 웹 공격) 같은 어플리케이션 계층 공격을 막을 수 없으므로, Application Gateway에 웹 애플리케이션 방화벽(WAF)을 두어 실시간으로 공격을 차단합니다. 정상 통과하지만, SQL 인젝션을 시도하면 차단되는 모습입니다. OWASP(오와스프) 3.2 룰셋을 차단 모드로 적용한 WAF가 공격 트래픽을 백엔드에 닿기 전에 막아냄이 검증되었습니다.

**슬라이드 5 — 보안 ① 외부 DB 직접 접근 차단**
DB는 온프레미스에 두고 VPN 터널로만 접근하도록 설계했기 때문에, DB 포트가 외부에 노출되어 있지 않은지 확인해야 합니다. 외부 인터넷에서 온프레미스 DB의 공인 경로(1.220.76.2:3306)로 접속을 시도하면 연결이 실패하는 모습입니다. DB가 IPsec 터널 내부에서만 접근 가능하고, 외부에는 포트가 닫혀 있음이 검증되었습니다.

**슬라이드 6 — 네트워크 ② Hub-Spoke VNet Peering**
허브-스포크 구조가 동작하려면 공유 서비스를 둔 허브와 워크로드를 둔 스포크가 실제로 연결되어 있어야 합니다. 허브 VNet과 스포크 VNet의 Peering(VNet 간 직접 연결) 상태가 Connected로 표시된 모습입니다. 양 리전 모두 정상 연결되어, 망 분리와 리전 내 고속 통신이 동시에 확보되었음이 검증되었습니다.

**슬라이드 7 — 고가용성 ③ 인스턴스 장애 시 서비스 지속**
서버가 한 대뿐이면 그 한 대의 장애가 곧 서비스 중단으로 이어지므로, VMSS로 여러 대를 운영해 단일 장애점(SPOF)을 제거했는지 확인했습니다. 인스턴스 2대 중 1대를 의도적으로 중지시킨 상태에서도 쇼핑몰이 정상 접속되는 모습입니다. 한 대가 중단되어도 남은 인스턴스가 서비스를 이어받아 무중단으로 동작함이 검증되었습니다.

**슬라이드 8 — 고가용성 ③ CPU 부하 Auto Scaling**
트래픽이 몰릴 때 수동으로 서버를 늘리면 대응이 늦으므로, 부하에 따라 인스턴스를 자동 증감하는 오토스케일이 동작하는지 확인합니다. CPU를 일부러 점유시키는 명령어(`yes`)로 CPU를 약 99%까지 올려 임계치 70%를 초과시키자, 인스턴스가 2대에서 5대로 자동 확장된 모습입니다. 사람 개입 없이 부하 변동에 자동 대응함이 검증되었습니다.

**슬라이드 9 — 하이브리드 ④ VPN 연결 상태**
DB를 온프레미스에 남긴 이상, 클라우드와 온프레미스를 안전하게 잇는 터널이 반드시 살아 있어야 합니다. Azure VPN Gateway와 온프레미스 Bluemax 방화벽 사이의 Site-to-Site IPsec 터널이 Connected 상태인 모습입니다. 양 끝단의 암호화 정책(AES256·SHA256·DHGroup14)을 일치시켜, 두 거점이 암호화된 전용 경로로 연결되어 있음이 검증되었습니다.

**슬라이드 10 — 하이브리드 ④ 온프레미스 DB 연동**
터널이 연결된 것과 실제로 DB 통신이 이루어지는 것은 별개이므로, 웹 서버가 온프레미스 DB를 실제로 읽을 수 있는지 확인합니다. VMSS에서 온프레미스 MySQL(10.10.34.119)에 접속해 DB 목록을 조회하는 명령어(`show databases`)를 실행하면 wordpress 데이터베이스가 조회되는 모습입니다. 클라우드 웹 계층과 온프레미스 데이터 계층이 VPN을 통해 정상 연동됨이 검증되었습니다.

**슬라이드 11 — 캐시 ⑤ Redis PE 격리 · 연동**
Redis(자주 쓰는 데이터를 메모리에 저장하는 캐시) 같은 관리형 서비스는 기본적으로 공인 접속점을 가지므로, Private Endpoint로 사설망에 격리되었는지 확인해야 합니다. VMSS에서 Redis 주소를 조회하는 명령어(`nslookup`)를 실행하면 공인 IP가 아닌 사설 IP(10.1.2.5)가 반환되고, WordPress에서도 `wp redis status`가 Connected로 표시된 모습입니다. Redis가 사설망에 격리된 채 정상 연동됨이 검증되었습니다.

**슬라이드 12 — 캐시 ⑤ Redis Cache Hits**
캐시를 도입했다면 실제로 캐시 적중이 일어나 DB 부하를 덜어 주어야 의미가 있습니다. Azure Monitor에서 Redis의 Cache Hits 지표가 누적되는 모습입니다. 반복 조회를 DB 대신 Redis가 처리하며, 응답 속도와 DB 부하가 개선되고 있음이 검증되었습니다.

**슬라이드 13 — 재해복구 ⑥ Traffic Manager 절체**
다중 리전의 목적은 리전 단위 장애에도 서비스를 유지하는 것이므로, Traffic Manager가 장애를 감지해 자동으로 트래픽을 절체하는지 확인합니다. 평상시 도메인 조회 시 Central IP(20.214.152.240)가 반환되다가, Central 엔드포인트를 비활성화하자 Japan이 Online으로 전환되며 조회 결과가 Japan IP(52.140.213.49)로 바뀐 모습입니다. 주 리전 장애를 감지해 DNS를 DR 리전으로 자동 절체함이 검증되었습니다.

**슬라이드 14 — 재해복구 ⑥ Japan 서비스**
트래픽이 Japan으로 넘어가도 정작 Japan 쪽 서비스가 뜨지 않으면 페일오버는 실패이므로, 절체 후 실제 서비스 가용성을 확인합니다. 절체된 상태에서 Japan East 리전의 쇼핑몰이 정상 접속되는 모습입니다. DR 리전이 단순 대기 상태가 아니라 즉시 서비스를 이어받을 수 있는 상태로 준비되어 있음이 검증되었습니다.

**슬라이드 15 — 재해복구 ⑥ 데이터 일관성 (1)**
페일오버에서 가장 위험한 것은 리전마다 데이터가 달라지는 것이므로, 회원가입 데이터가 리전과 무관하게 한곳에 모이는지 확인합니다. 먼저 Central에서 test01 계정으로 회원가입을 수행한 모습입니다. 이 데이터가 어디에 저장되는지는 다음 슬라이드에서 확인합니다.

**슬라이드 16 — 재해복구 ⑥ 데이터 일관성 (2)**
앞에서 생성한 데이터가 온프레미스 단일 DB에 저장되는지가 핵심입니다. Central에서 가입한 직후, 온프레미스 MySQL의 shop_users 테이블에서 test01 레코드가 그대로 조회되는 모습입니다. DB가 온프레미스 한곳에 모여 있어 어느 리전에서 접속하든 동일한 데이터가 보장되며, 페일오버가 데이터 정합성에 영향을 주지 않음이 검증됩니다.

**슬라이드 17 — 공유 스토리지 ⑦ Azure Files 인스턴스 간 공유**
VMSS는 인스턴스가 수시로 증감하므로, 업로드 파일을 로컬 디스크에 두면 인스턴스마다 내용이 달라집니다. 따라서 업로드 폴더를 공유 스토리지로 묶었는지 확인합니다. 업로드 폴더(wp-content/uploads)를 Azure Files로 마운트한 뒤, 인스턴스 A에서 올린 sharetest.txt를 인스턴스 B에서 열면 동일한 'test from A' 내용이 그대로 조회되는 모습입니다. 모든 인스턴스가 같은 파일을 공유하며, 스케일아웃으로 생성되는 인스턴스도 동일 볼륨을 마운트함이 검증되었습니다.

**슬라이드 18 — 모니터링 ⑧ Log Analytics**
인프라 상태를 가시화하지 않으면 장애 대응이 늦어지므로, 핵심 리소스의 로그와 지표가 한곳에 수집되는지 확인합니다. Log Analytics에서 VMSS·Application Gateway·Firewall의 메트릭과 로그가 수집되고 있는 모습입니다. 문제 발생 시 추적·대응이 가능한 운영 가시성이 확보되었음이 검증되었습니다.

이상 8개 영역 모두 설계 의도대로 동작함을 실제 명령 출력과 포털 화면으로 검증 완료했습니다.

**슬라이드 19 — 08 결론 · 한계점 및 개선 방향**
이번 구축은 학습·검증 환경 기준으로 목표를 달성했지만, 실제 운영에 올리려면 보완할 점이 분명히 있습니다. 한계와 개선 방향을 정리하겠습니다. 보안 측면에서는 리스너가 HTTP 80이라 HTTPS가 적용되지 않았고, 비밀번호가 평문으로 들어가 있으며, 스토리지는 Access Key 인증입니다. 각각 443 리스너와 Key Vault SSL, Key Vault와 Managed Identity, Entra ID·SAS·키 회전으로 개선할 수 있습니다. 가용성 측면에서는 온프레미스 DB가 단일 인스턴스이고 VPN이 단일 터널이라 단일 장애점이 남아 있습니다. DB 이중화나 읽기 복제본, Active-Active VPN·ExpressRoute로 보완 가능하고, Traffic Manager는 DNS 기반이라 절체 지연이 있어 Front Door로 즉시 절체를 고려할 수 있습니다. 운영 측면에서는 CI/CD가 없어 수동 배포 중이므로, 파이프라인을 구축하면 코드 변경을 자동 배포할 수 있습니다. 향후에는 Front Door 기반 글로벌 가속, AKS·Serverless 전환, 데이터 분석 연동, Sentinel 기반 보안 자동화로 확장할 수 있습니다. 이상으로 검증 결과와 결론 발표를 마치겠습니다.
