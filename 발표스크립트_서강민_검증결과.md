## 슬라이드 1 — 보안 ① VMSS 공인 IP 미할당 · Firewall 강제 경유

**[목적]** 제로 트러스트 설계의 출발점은 웹 서버를 외부에 직접 노출하지 않는 것입니다. 웹 서버가 공인 IP를 가지면 공격자가 Application Gateway를 우회해 직접 접근할 수 있기 때문에, VMSS는 공인 IP 없이 게이트웨이 뒤에만 두어야 합니다.

**[화면]** 실제로 VMSS 인스턴스에는 공인 IP가 할당되어 있지 않습니다. 그리고 VMSS 안에서 `curl ifconfig.me`를 실행하면, 인스턴스 IP가 아니라 Firewall 공인 IP인 20.214.183.226이 반환됩니다.

**[결론]** 인바운드는 게이트웨이로만, 아웃바운드는 전부 Firewall을 거쳐 나간다는 뜻입니다. 외부 노출 경로를 양방향으로 차단한 것입니다.

## 슬라이드 2 — 보안 ① UDR 라우트 테이블 (Central)

**[목적]** 방금 본 'Firewall 강제 경유'는 저절로 되는 게 아니라 UDR(사용자 지정 라우팅)로 강제해야 합니다. 기본 상태로 두면 트래픽이 인터넷으로 바로 나가 Firewall을 우회하기 때문입니다.

**[화면]** Central Spoke의 Web 서브넷과 PE 서브넷에 라우트 테이블을 걸어, 0.0.0.0/0 모든 경로가 Firewall(10.0.0.4)로 향하도록 지정했습니다.

**[결론]** 이 설정 때문에 앞 슬라이드의 아웃바운드 강제 경유가 성립합니다. 모든 외부 통신이 중앙 Firewall 한 곳에서 통제됩니다.

## 슬라이드 3 — 보안 ① UDR 라우트 테이블 (Japan)

**[목적]** DR 리전이 주 리전과 보안 수준이 다르면, 페일오버 순간 보안 구멍이 생깁니다. 그래서 Japan East에도 Central과 동일한 라우팅을 적용했습니다.

**[화면]** Japan Web·PE 서브넷의 0.0.0.0/0도 Japan Firewall(10.2.0.4)로 향합니다.

**[결론]** 양 리전이 완전히 대칭이라, 어느 리전으로 절체되더라도 동일한 보안 정책이 유지됩니다.

## 슬라이드 4 — 보안 ① WAF 정상/차단 응답

**[목적]** 네트워크 계층 차단만으로는 SQL Injection 같은 애플리케이션(L7) 공격을 막을 수 없습니다. 그래서 Application Gateway에 WAF를 두어 웹 공격을 실시간으로 걸러냅니다.

**[화면]** WAF_v2에 OWASP 3.2 룰셋을 Prevention 모드로 적용했습니다. 정상 요청은 200 OK로 통과하지만, SQL Injection 패턴이 들어오면 403 Forbidden으로 차단됩니다.

**[결론]** 실제 공격 트래픽이 백엔드에 닿기 전에 게이트웨이에서 차단됨을 확인했습니다.

## 슬라이드 5 — 보안 ① 외부 DB 직접 접근 차단

**[목적]** DB는 온프레미스에 두고 VPN 터널로만 접근하도록 설계했습니다. 만약 DB 포트가 외부에 열려 있다면 하이브리드 구조 자체가 무의미해지므로, 외부 노출이 없는지 확인해야 합니다.

**[화면]** 외부 인터넷에서 온프레미스 DB의 공인 경로(1.220.76.2:3306)로 접속을 시도하면 실패합니다.

**[결론]** DB는 오직 IPsec 터널 내부에서만 접근 가능하고, 외부에는 포트가 닫혀 있음을 검증했습니다.

## 슬라이드 6 — 네트워크 ② Hub-Spoke VNet Peering

**[목적]** Hub-Spoke 구조가 의미를 가지려면, 공유 서비스를 둔 Hub와 워크로드를 둔 Spoke가 실제로 연결되어 있어야 합니다. 그 연결 상태를 확인합니다.

**[화면]** 각 리전에서 Hub VNet과 Spoke VNet을 Peering으로 연결했고, Central·Japan 모두 상태가 Connected입니다.

**[결론]** Azure 백본망을 통해 Hub-Spoke가 정상 통신하며, 망 분리와 확장성을 동시에 확보했습니다.

## 슬라이드 7 — 고가용성 ③ 인스턴스 장애 시 서비스 지속

**[목적]** 단일 서버 구조는 한 대만 죽어도 서비스가 멈춥니다. VMSS로 여러 대를 운영하는 이유가 바로 이 단일 장애점(SPOF) 제거인데, 실제로 그렇게 동작하는지 확인합니다.

**[화면]** 인스턴스 2대를 운영하다가 1대를 의도적으로 중지시켰습니다. 그 상태에서도 쇼핑몰은 정상 접속됩니다.

**[결론]** 인스턴스 한 대가 죽어도 남은 인스턴스가 서비스를 이어받아, 무중단 서비스가 보장됩니다.

## 슬라이드 8 — 고가용성 ③ CPU 부하 Auto Scaling

**[목적]** 트래픽이 몰릴 때 수동으로 서버를 늘리면 늦습니다. 부하에 따라 자동으로 증감하도록 오토스케일을 걸어 두었는데, 임계치를 넘기면 정말 확장되는지 검증합니다.

**[화면]** `yes` 명령으로 CPU를 약 99%까지 올려 임계치 70%를 초과시켰습니다. 그러자 인스턴스가 2대에서 5대로 자동 확장됩니다.

**[결론]** 사람 개입 없이 부하 변동에 자동 대응함을 확인했습니다. 성능과 비용을 동시에 잡는 구조입니다.

## 슬라이드 9 — 하이브리드 ④ VPN 연결 상태

**[목적]** DB를 온프레미스에 남기는 이유는 데이터 주권 때문이고, 그러려면 Azure와 온프레미스를 안전하게 잇는 터널이 반드시 살아 있어야 합니다.

**[화면]** Azure VPN Gateway와 온프레미스 Bluemax 방화벽 사이의 Site-to-Site IPsec 터널이 Connected 상태입니다. 양 끝단의 IKE/IPsec 정책을 AES256·SHA256·DHGroup14로 정확히 일치시켜 터널을 성립시켰습니다.

**[결론]** 클라우드와 온프레미스가 암호화된 전용 경로로 연결되어 있음을 확인했습니다.

## 슬라이드 10 — 하이브리드 ④ 온프레미스 DB 연동

**[목적]** 터널이 연결된 것과 실제로 DB 쿼리가 오가는 것은 다릅니다. 웹 서버가 온프레미스 DB를 실제로 읽고 쓸 수 있는지 확인합니다.

**[화면]** VMSS에서 온프레미스 MySQL(10.10.34.119)에 접속해 `show databases`를 실행하면 wordpress 데이터베이스가 조회됩니다.

**[결론]** 클라우드 웹 계층과 온프레미스 데이터 계층이 VPN 너머로 정상 연동됨을 검증했습니다.

## 슬라이드 11 — 캐시 ⑤ Redis PE 격리 · 연동

**[목적]** Redis 같은 PaaS는 기본적으로 공인 엔드포인트를 갖기 때문에, 그대로 두면 외부 노출 위험이 있습니다. Private Endpoint로 사설망에 격리했는지부터 확인해야 합니다.

**[화면]** VMSS에서 Redis를 nslookup하면 공인 IP가 아니라 사설 IP 10.1.2.5가 반환됩니다. 그리고 WordPress에서 `wp redis status`가 Connected로 나옵니다.

**[결론]** Redis가 사설망에 격리된 채로 WordPress와 정상 연동됨을 확인했습니다.

## 슬라이드 12 — 캐시 ⑤ Redis Cache Hits

**[목적]** 캐시를 붙였다면 실제로 적중이 일어나 DB 부하를 줄여 주어야 의미가 있습니다.

**[화면]** Azure Monitor에서 Redis Cache Hits가 누적되는 그래프를 확인할 수 있습니다.

**[결론]** 캐시가 실제로 동작하며, 반복 조회를 DB 대신 Redis가 처리해 응답 속도와 DB 부하를 개선하고 있습니다.

## 슬라이드 13 — 재해복구 ⑥ Traffic Manager 절체

**[목적]** 다중 리전을 둔 핵심 이유는 리전 단위 장애에도 서비스를 유지하기 위해서입니다. Traffic Manager가 장애를 감지해 자동으로 트래픽을 넘기는지 검증합니다.

**[화면]** 우선순위 라우팅으로 Central이 1순위 Active, Japan이 2순위 Standby입니다. 정상일 때는 도메인 조회 시 Central IP(20.214.152.240)가 반환됩니다. Central 엔드포인트를 비활성화하자 Japan이 Online으로 바뀌고, 조회 결과가 Japan IP(52.140.213.49)로 전환됩니다.

**[결론]** 주 리전 장애를 프로브로 감지해 DNS를 자동으로 DR 리전으로 넘기는 것을 확인했습니다.

## 슬라이드 14 — 재해복구 ⑥ Japan 서비스

**[목적]** 트래픽이 Japan으로 넘어가도, 정작 Japan 쪽 서비스가 안 뜨면 페일오버는 실패입니다. 절체 후 실제 서비스가 살아 있는지 확인합니다.

**[화면]** 절체된 상태에서 Japan East 리전의 쇼핑몰이 정상 접속됩니다.

**[결론]** DR 리전이 단순 대기 상태가 아니라, 즉시 서비스를 이어받을 수 있는 상태로 준비되어 있음을 검증했습니다.

## 슬라이드 17 — 공유 스토리지 ⑦ Azure Files 인스턴스 간 공유

**[목적]** VMSS는 인스턴스가 늘었다 줄었다 하기 때문에, 업로드 파일을 각 인스턴스 로컬 디스크에 두면 인스턴스마다 파일이 달라집니다. 그래서 업로드 폴더를 공유 스토리지로 묶어야 합니다.

**[화면]** wp-content/uploads를 Azure Files로 마운트했습니다. 인스턴스 A에서 sharetest.txt를 업로드하고, 인스턴스 B에서 열면 동일한 "test from A" 내용이 그대로 보입니다.

## 슬라이드 19 — 08 결론 · 한계점 및 개선 방향

**[목적]** 이번 구축은 학습·검증 환경 기준으로 목표를 달성했지만, 실제 운영에 올리려면 보완할 점이 분명히 있습니다. 솔직하게 한계와 개선 방향을 정리하겠습니다.

**보안** — 리스너가 HTTP 80이라 HTTPS가 적용되지 않았고, 비밀번호가 평문으로 들어가 있으며, 스토리지는 Access Key 인증입니다. 각각 443 리스너+Key Vault SSL, Key Vault+Managed Identity, Entra ID·SAS·키 회전으로 개선할 수 있습니다.

**가용성** — 온프레미스 DB가 단일 인스턴스이고 VPN이 단일 터널이라 단일 장애점이 남아 있습니다. DB 이중화·읽기 복제본, Active-Active VPN·ExpressRoute로 보완 가능합니다. 또 Traffic Manager는 DNS 기반이라 TTL(30초)만큼 절체 지연이 있어, Front Door(L7) 도입으로 즉시 절체를 고려할 수 있습니다.

**운영** — CI/CD가 없어 Terraform을 수동으로 apply하고 있습니다. 파이프라인을 구축하면 코드 변경을 자동 배포할 수 있습니다.

**향후 확장** — Front Door+WAF 기반 글로벌 가속, AKS·Serverless 클라우드 네이티브 전환, 데이터 인텔리전스, Sentinel 기반 보안 자동화로 확장할 수 있습니다. 이상으로 검증 결과와 결론 발표를 마치겠습니다.

## ⚠ 발표 전 확인 (슬라이드 캡션 1건)

슬라이드 2(Central UDR) 캡션이 `Firewall(10.2.0.4)`로 되어 있는데, 10.2.0.4는 **Japan** Firewall 사설 IP입니다. 보고서 기준 **Central Firewall은 10.0.0.4**입니다. 캡션을 `10.0.0.4`로 고치거나, 말할 때 Central은 10.0.0.4로 설명하시길 권합니다. (슬라이드 3 Japan의 10.2.0.4는 정확합니다.)
