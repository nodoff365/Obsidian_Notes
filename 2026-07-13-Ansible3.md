## 1. Module

> Shell 모듈로 직접 명령어를 실행할 수도 있지만, Ansible이 제공하는 내장 모듈을 사용하는 것이 낫다. 모듈은 멱등성(idempotency)을 보장해서, 같은 플레이북을 여러 번 실행해도 시스템 상태가 일관되게 유지된다. 따라서 가능하면 shell 모듈보다는 해당 기능을 담당하는 모듈을 우선적으로 사용한다.

---

## 2. Ad-hoc: 파일/디렉토리 관리 (file 모듈)

> file 모듈을 사용하면 파일/디렉토리 생성/삭제/권한 변경을 할 수 있다. shell 모듈과 달리 동일한 경로에 대해 여러 번 실행해도 에러가 나지 않는다.

### 2.1 파일/디렉토리 생성

```bash
ansible node -m file -a "path=/test state=directory"
```

![](/assets/images/Ansible/3_1.png)

- `state=directory`는 디렉토리 생성. 
- `state=touch`면 빈 파일 생성. 이미 존재해도 에러 없음 (멱등성).

### 2.2 파일/디렉토리 삭제

```bash
ansible node -m file -a "path=/test state=absent"
```

![](/assets/images/Ansible/3_2.png)

- `state=absent`은 존재하는 파일/디렉토리 삭제. 이미 없는 경로라도 에러가 나지 않음 (멱등성).

## 3. Ad-hoc: 사용자 관리(Shell 모듈)

> Shell 모듈을 사용해 useradd/userdel 명령어로 사용자를 생성/삭제하는 방식이다. 직접적이지만 멱등성이 없어서 여러 번 실행하면 에러가 난다.

### 3.1 Shell 모듈로 사용자 생성

```bash
ansible node -m shell -a "useradd a && echo 'It1' | passwd --stdin a"
ansible node -m shell -a "tail -5 /etc/passwd"
```

![](/assets/images/Ansible/3_3.png)

- `useradd`로 사용자 생성하고, `passwd --stdin`으로 비밀번호를 설정한다.
### 3.2 Shell 모듈로 사용자 삭제

```bash
ansible node -m shell -a "userdel -r a"
```

![](/assets/images/Ansible/3_4.png)

- `-r` 옵션으로 홈 디렉토리(`/home/사용자명`)와 메일 스풀(`/var/spool/mail/사용자명`)까지 함께 삭제한다.

## 4. Ad-hoc: 사용자 관리(User 모듈)

> User 모듈을 사용하면 사용자 생성/삭제/비밀번호 설정을 상태 기반으로 처리한다. 멱등성이 있어서 같은 명령을 여러 번 실행해도 안전하다.

### 4.1 User 모듈로 사용자 생성

```bash
ansible node -m user -a "name=b state=present"
```

![](/assets/images/Ansible/3_5.png)

- `state=present`은 사용자가 존재해야 한다는 상태다. 이미 있으면 변경 없이 넘어간다 (멱등성).

### 4.2 User 모듈로 비밀번호 설정

```bash
ansible node -m user -a "name=b password='It1'"

ansible node -m shell -a "tail -2 /etc/shadow"
```

![](/assets/images/Ansible/3_6_1.png)
![](/assets/images/Ansible/3_6_2.png)

- `password` 파라미터는 자동 암호화되지 않고, 넘긴 값이 그대로 `/etc/shadow`에 저장된다. 평문(`It1`)을 넣으면 평문 그대로 저장되고, `password_hash('sha512')` 필터로 미리 암호화한 값을 넣으면 해시로 저장된다. 안전하게 설정하려면 항상 해시값을 넘겨야 한다.

### 4.3 User 모듈로 사용자 삭제

```bash
ansible node -m user -a "name=a state=absent remove=true"
```

![](/assets/images/Ansible/3_7.png)

- `state=absent`은 사용자 삭제, `remove=true`는 홈디렉토리까지 함께 삭제한다.

### 4.4 Shell 모듈 vs User 모듈 비교

| 항목 | Shell 모듈 | User 모듈 |
|---|---|---|
| 멱등성 | 없음 (여러 번 실행 시 에러) | 있음 (여러 번 실행해도 안전) |
| 사용 방식 | 직접 명령어 입력 | 선언적 상태 지정 |
| 권장 | shell 명령어가 필요한 경우만 | 항상 |

## 5. Playbook: 파일/디렉토리 관리 (test.yml)

> Playbook을 사용하면 여러 작업을 순서대로 정의해서 재사용할 수 있다. file 모듈로 파일/디렉토리를 생성/삭제한다.

### 5.1 생성 플레이북

```yaml
---
- name: make file /test
  hosts: node
  gather_facts: false
  ignore_errors: true 
  
  tasks:
  - name: make file test
    file: 
      path: /test
      state: touch
      mode: '0777'

  - name: make directory /tdirec
    file:
      path: /tdirec
      state: directory
      mode: '0777'
```

```bash
ansible-playbook test.yml

ansible node -m shell -a "ls -al /test /tdirec"
```

![](/assets/images/Ansible/3_8_1.png)
![](/assets/images/Ansible/3_8_2.png)
- `mode: '0777'`은 권한을 8진법으로 설정한다. 문자열로 감싸야 YAML에서 수치로 해석되지 않는다.

### 5.2 삭제 플레이북

```yaml
---
- name: delete file & directory
  hosts: node
  gather_facts: false
  ignore_errors: true

  tasks:
  - name: delete file test
    file:
      path: /test
      state: absent

  - name: delete directory /tdirec
    file:
      path: /tdirec
      state: absent
```

```bash
ansible-playbook test.yml
ansible node -m shell -a "ls -al /"
```

**캡처 위치**: 9

## 6. Playbook: 사용자 관리 및 비밀번호 암호화 (user.yml)

> Playbook에서는 명시적으로 `password_hash` 필터를 사용해서 평문 비밀번호를 sha512 해시값으로 변환한다. Ad-hoc과 달리 보안이 더 강화된다.

```yaml
---
- name: create user with password
  hosts: web
  gather_facts: false
  ignore_errors: true

  tasks:
  - name: create user a
    user:
      name: a
      state: present
      password: "{{ 'It1' | password_hash('sha512') }}"
      update_password: always
```

```bash
ansible-playbook user.yml
ansible web -m shell -a "tail -5 /etc/shadow"
```

**캡처 위치**: 10

> `{{ 'It1' | password_hash('sha512') }}`는 Jinja2 필터 문법으로, 플레이북 실행 시 평문 비밀번호가 sha512 해시값으로 변환된다.

## 7. Playbook: 과제 - DB 호스트 사용자 및 파일 설정 (work.yml)

> 특정 호스트(db, 10.0.0.14)에 대해 사용자 생성, uid/gid 명시, 파일/디렉토리 권한 설정을 한 번에 처리한다.

```yaml
---
- name: DB host user and file setup
  hosts: db
  gather_facts: false
  ignore_errors: true
  
  tasks:
  - name: create user a (uid=2000)
    user:
      name: a
      uid: 2000
      password: "{{ 'lt1' | password_hash('sha512') }}"
      state: present

  - name: create user b (uid=3000)
    user:
      name: b
      uid: 3000
      password: "{{ 'lt1' | password_hash('sha512') }}"
      state: present

  - name: create /test file
    file:
      path: /test
      state: touch
      owner: a
      group: b
      mode: 0777

  - name: create /tbabo directory
    file:
      path: /tbabo
      state: directory
      owner: a
      group: b
      mode: 0777
```

```bash
ansible-playbook work.yml
ansible db -m shell -a "tail -2 /etc/passwd"
ansible db -m shell -a "ls -ld /test /tbabo"
```

**캡처 위치**: 11 (/etc/passwd에서 uid 확인) / 12 (ls -ld에서 owner:group과 권한 확인)

> `owner`, `group` 옵션으로 소유자와 그룹을 지정한다. file 모듈이 자동으로 권한을 적용한다.

## 8. Playbook: 반복문으로 여러 항목 삭제 (rmwork.yml)

> `with_items`은 플레이북에서 반복문을 구현하는 방식이다. `{{ item }}`은 매 반복마다 리스트 항목을 가리킨다. 이렇게 하면 동일한 태스크를 여러 번 정의할 필요 없이 한 번에 처리할 수 있다.

```yaml
---
- name: delete user & file
  hosts: db
  gather_facts: false
  ignore_errors: true

  tasks:
  - name: delete user
    user:
      name: "{{ item }}"
      state: absent
      remove: yes
    with_items:
      - a
      - b

  - name: delete file
    file:
      path: "{{ item }}"
      state: absent
    with_items:
      - /test
      - /tbabo
```

```bash
ansible-playbook rmwork.yml
```

**캡처 위치**: 13

> 첫 번째 `with_items`은 `item = a`, `item = b`로 2번 반복해서 두 사용자를 순서대로 삭제한다. 두 번째 `with_items`도 마찬가지로 2개 경로를 순서대로 삭제한다.
