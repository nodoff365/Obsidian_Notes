# Azure 인프라 초보자 설명서

> 비유 → 없으면 어떻게 되는지 → 왜 이걸 선택했는지 순서로 설명합니다.
> 복잡한 것일수록 더 자세히 설명합니다.

---

## 1. 이 인프라가 뭘 하는 건가요?

WordPress로 만든 웹사이트를 사용자들이 끊김 없이, 안전하게 이용할 수 있도록 Azure 클라우드 위에 구성한 서버 환경입니다.

쉽게 말하면 웹사이트를 운영하기 위한 **건물 + 보안 시스템 + 비상구**를 만드는 것입니다.

| 비유 | 실제 |
|---|---|
| 건물 | 웹 서버 (VM, VMSS) |
| 건물 정문 보안 검색대 | Application Gateway + WAF |
| 건물 내부 CCTV + 차단기 | Azure Firewall |
| 경비원 전용 통로 | Azure Bastion |
| 직원 전용 뒷문 (출입 전용) | NAT Gateway |
| 금고 | MySQL DB, Storage |
| 비상구 | DR 리전 (Korea South) |
| 본사-지사 전용 암호 전화선 | VPN Gateway |
| 사내 전화번호부 | Private DNS Zone |

---

## 2. Hub-Spoke 구조란?

**비유: 회사 건물 구조**

회사에서 보안실(Hub)을 건물 입구에 두고, 실제 직원들이 일하는 사무실(Spoke)은 따로 두는 구조입니다.

```
인터넷
  ↓
[Hub VNet] ← 보안 장비 모음 (Firewall, App Gateway, VPN Gateway)
  ↓ Peering (Azure 내부 전용 고속도로, 인터넷 미경유)
[Spoke VNet] ← 실제 서비스 (Web 서버, DB, Storage)
```

**없으면 어떻게 되나요?**

모든 서버가 하나의 네트워크에 섞여 있으면 Web 서버가 해킹당했을 때 DB까지 바로 접근 가능합니다. Hub-Spoke로 분리하면 각 영역마다 별도 방어선이 생겨서 한 곳이 뚫려도 다른 곳은 안전합니다.

**왜 1단계부터 Hub-Spoke를 잡았나요?**

나중에 Firewall, VPN을 추가할 때 구조 변경 없이 리소스만 추가하면 됩니다. 처음부터 안 잡으면 4단계에서 전체를 다시 설계해야 합니다.

---

## 3. 단계별 개요

### 1단계 — 뼈대
최소 구성으로 웹사이트가 동작하는지 확인합니다.
- 사용자 → App Gateway → Web VM → MySQL
- 관리자 → Bastion → Web VM
- Web VM 아웃바운드 → NAT Gateway → 인터넷

### 2단계 — 이중화
서버·DB 하나가 죽어도 서비스가 계속되도록 만듭니다.
- VM 1대 → VMSS (여러 대 자동 확장)
- MySQL 단일 → Zone HA + Read Replica

### 3단계 — 보안 강화
외부 공격을 막고 실시간으로 모니터링합니다.
- WAF 활성화, Redis 세션 캐시, Azure Monitor

### 4단계 — 하이브리드 + DR
두 리전 운영, 온프레미스 서버 연결, 중앙 보안 통제.
- Azure Firewall, VPN Gateway, Korea South DR

### Application Gateway + WAF

**비유: 건물 정문 + 보안 검색대**

모든 사용자 요청이 이곳을 통과합니다. WAF(Web Application Firewall)는 해킹 시도를 자동으로 탐지하고 차단합니다.

**없으면 어떻게 되나요?**

사용자 요청이 서버로 바로 들어가고, SQL Injection(DB를 직접 공격하는 해킹 기법), XSS(악성 스크립트를 심는 기법) 같은 공격을 막을 방법이 없습니다.

**Azure Load Balancer 대신 쓰는 이유**

Load Balancer는 L4 장비라 트래픽을 숫자(포트, IP)로만 분산합니다. Application Gateway는 L7 장비라 요청 내용을 직접 분석해서 공격을 막을 수 있습니다.

**WAF Detection vs Prevention — 왜 Detection부터 시작하나요?**

```
Detection 모드:
공격 감지 → 그냥 통과 + 로그만 기록
→ 실제로 막지 않음
→ 오탐(정상 트래픽을 공격으로 오인) 여부 확인용

Prevention 모드:
공격 감지 → 즉시 차단
→ 실제 방어
→ 오탐 있으면 정상 사용자도 막힘
```

WordPress 플러그인 중에 WAF가 공격으로 오인하는 요청을 보내는 게 있습니다. 바로 Prevention으로 시작하면 플러그인이 작동 안 할 수 있어요. Detection으로 로그를 먼저 보고 이상 없으면 Prevention으로 전환합니다.

**App Gateway가 Firewall보다 앞에 있는 이유**

App Gateway v2는 Azure 내부 관리 목적으로 공인 IP와 인터넷 직접 통신이 반드시 필요합니다. Firewall 뒤에 두면 두 가지 문제가 생깁니다.

```
문제 1 — 비대칭 라우팅:
Firewall → AppGW로 요청 보냄
AppGW → 응답을 Firewall 안 거치고 직접 반환
Firewall: "내가 보낸 요청이 아닌데?" → 패킷 드롭 → 연결 끊김

문제 2 — 관리 트래픽 차단:
App Gateway는 Azure가 내부적으로 관리·업데이트하기 위해 인터넷과 직접 통신 필요
Firewall 뒤에 두면 이 트래픽이 막혀 App Gateway 오작동
```

### Azure Bastion vs Jumpbox VM

**관리자가 서버 내부에 접속하는 두 가지 방법**

**Jumpbox VM (쓰지 않는 이유)**

```
관리자
  ↓ SSH (22번 포트, 인터넷 전체에 공개됨)
Jumpbox VM (공인 IP 있음)
  ↓ SSH (22번)
Web VM
```

22번 포트가 인터넷에 열려 있어서 전 세계에서 자동화된 해킹 도구가 아이디/비밀번호를 수만 번 시도(무차별 대입 공격)할 수 있습니다.

**Azure Bastion (선택한 이유)**

```
관리자
  ↓ HTTPS (443번 포트, 브라우저로 접속)
Azure Bastion
  ↓ (내부적으로 22번, 외부에는 절대 노출 안 됨)
Web VM
```

브라우저에서 443번으로 접속하면 Azure가 알아서 VM에 연결해줍니다. 22번 포트를 인터넷에 열 필요가 없어서 무차별 대입 공격 자체가 불가능합니다.

**AzureBastionSubnet 이름과 /26 크기가 고정인 이유**

Azure 시스템이 이 이름과 크기를 보고 "아, 여기가 Bastion 전용 공간이구나"라고 인식합니다. 이름이 다르거나 /27보다 작으면 Bastion 배포 자체가 실패합니다.

### NAT Gateway

**비유: 건물 직원 전용 뒷문 (출구만, 입구 없음)**

내부 서버가 소프트웨어 업데이트, 외부 API 호출 등을 위해 인터넷으로 나갈 때 사용합니다. 반대로 외부에서 이 문으로 들어오는 것은 원천 차단됩니다.

**없으면 어떻게 되나요?**

서버가 인터넷에 전혀 접근 못 해서 소프트웨어 업데이트, 패키지 설치가 불가능합니다.

**왜 4단계에서 제거하나요? — NAT GW + UDR 충돌 설명**

4단계에서 Web-Subnet에 UDR을 적용합니다. UDR은 "모든 트래픽은 무조건 Firewall로 가라"는 강제 규칙입니다.

```
Azure 공식 문서에 따른 우선순위:
UDR → NAT Gateway → 공인 IP → 로드밸런서 → 기본 경로
```

UDR이 NAT Gateway보다 우선순위가 높아서, UDR을 적용하면 NAT Gateway가 있어도 실제로 동작하지 않습니다. 결과적으로:

```
4단계 이후:
Web VM 아웃바운드
  ↓ UDR (강제)
Azure Firewall (Firewall이 SNAT으로 아웃바운드 처리)
  ↓
인터넷

→ NAT Gateway는 켜져 있어도 쓸모 없음
→ 비용($32/월)만 나가는 상태
→ 제거해서 비용 절감
```

### VMSS (Virtual Machine Scale Set)

**비유: 손님이 많아지면 직원을 자동으로 더 부르는 식당**

평소에는 서버 2대로 운영하다가 접속자가 많아지면 자동으로 서버를 늘리고, 줄면 다시 줄입니다.

```
CPU 70% 이상 유지 → 서버 1대 자동 추가
CPU 20% 미만 유지 → 서버 1대 자동 제거
AZ-1, AZ-2 두 곳 분산 → 한 곳 장애 시 다른 쪽에서 서비스 유지
```

**없으면 어떻게 되나요?**

트래픽이 갑자기 몰리면 서버가 버티지 못해 사이트가 다운됩니다. 한가할 때도 서버를 계속 켜둬야 해서 비용이 낭비됩니다.

**Compute Gallery가 필요한 이유**

VMSS가 서버를 자동으로 늘릴 때, 새 서버에 WordPress를 처음부터 설치하면 몇 분이 걸립니다. Gallery에 미리 완성된 이미지(골든 이미지)를 저장해두면 새 서버가 켜지는 즉시 바로 서비스 투입이 가능합니다.

**Session Affinity — 2단계에서 켜고 3단계에서 끄는 이유**

WordPress에서 로그인하면 "이 사람이 로그인했다"는 정보를 서버에 저장합니다. VMSS는 서버가 여러 대라서 매번 다른 서버로 연결되면 로그인 정보가 없는 서버로 가서 갑자기 로그아웃되는 현상이 생깁니다.

```
2단계 해결책 (임시):
Session Affinity 활성화
→ 같은 사람은 항상 같은 서버로 보냄
→ 근데 그 서버가 죽으면 그 사람 세션이 날아감

3단계 해결책 (완전):
Redis가 세션 정보를 중앙에서 관리
→ 어느 서버로 가도 Redis에서 꺼내오면 됨
→ Session Affinity 불필요 → 비활성화
```

### MySQL Flexible Server (VNet Integration)

**비유: 건물 내부 전용 금고 (외부에서 열쇠 없으면 접근 불가)**

WordPress의 게시글, 사용자 정보, 설정 등을 저장합니다. VNet 내부에서만 접근 가능하고 인터넷에서는 직접 접근이 불가능합니다.

**VNet Integration 방식이란?**

```
Private Endpoint 방식:
어디서든 배포 가능 → PE 리소스를 따로 만들어서 VNet에 연결

VNet Integration 방식 (우리가 쓰는 방식):
MySQL 서버가 처음부터 위임된 서브넷 안에 직접 배치됨
→ MySQL-Subnet이 MySQL 전용으로 통째로 위임
→ 다른 리소스는 이 서브넷에 절대 못 들어옴
```

**MySQL-Subnet과 PE-Subnet을 반드시 분리하는 이유**

위임 서브넷은 MySQL만을 위한 전용 공간입니다. Storage나 Redis의 Private Endpoint를 같은 서브넷에 두면 Azure가 "이 서브넷에 MySQL이 아닌 게 있네?"라고 오류를 냅니다.

**Zone HA — 자동 Failover 구조**

```
AZ-1 (주 서버) 장애 발생
        ↓ 자동 감지 (수십 초)
AZ-2 (대기 서버) 자동으로 주 서버로 승격
        ↓
서비스 계속 (다운타임 최소화)
```

**Read Replica**

게시글 조회(SELECT)처럼 읽기 요청은 Replica 서버가 처리합니다. 주 서버는 쓰기(INSERT, UPDATE)만 담당해서 부하가 크게 줄어듭니다.

### Redis Cache

**비유: 자주 보는 서류를 금고 대신 책상 위에 올려두기**

로그인 세션 정보를 빠른 메모리에 저장해서 매번 DB에 묻지 않아도 됩니다.

**Redis와 MySQL의 관계 — 자주 헷갈리는 부분**

Redis는 MySQL 앞에서 트래픽을 막는 게 아닙니다. 병렬로 연결됩니다.

```
잘못된 이해:
Web → Redis → MySQL (X)

올바른 구조:
Web ├→ Redis  (세션·로그인 정보)
    └→ MySQL  (게시글·상품 데이터)
```

**없으면 어떻게 되나요?**

매번 로그인 확인을 위해 MySQL에 쿼리를 날려야 해서 DB 부하가 커지고, 응답 속도가 느려집니다.

### NSG (Network Security Group)

**비유: 각 층마다 설치된 출입 카드 리더기**

서브넷 단위로 적용되는 방화벽입니다. 허용된 트래픽만 통과하고 나머지는 모두 차단합니다.

**기본 규칙**

```
Web-Subnet:
  - AppGW에서 80번 포트만 허용
  - AzureBastionSubnet에서 22번(SSH)만 허용
  - 나머지 전부 차단

MySQL-Subnet:
  - Web-Subnet에서 3306번 포트만 허용
  - 나머지 전부 차단
```

**2단계에서 NSG를 정교화하는 이유**

1단계에서는 "Bastion에서 오는 22번"이라고 막연하게 설정합니다. 2단계에서 Source를 `AzureBastionSubnet (10.1.0.0/26)`으로 명시해서 정확히 Bastion 서브넷에서만 SSH 접속을 허용합니다. 인터넷 전체(0.0.0.0/0)에서 22번이 열리는 것을 원천 차단합니다.

### Azure Firewall

**비유: 건물 내부 모든 통로의 CCTV + 자동 차단기**

4단계에서 배포됩니다. App Gateway가 받은 트래픽이 실제 서버로 가기 전에 Firewall을 통과합니다.

**트래픽 흐름**

```
인터넷
  ↓
App Gateway + WAF (공격 차단, L7 분석)
  ↓
Azure Firewall (내부 트래픽 정책 제어)
  ↓
VMSS (Web 서버)
  ↓
VPN Gateway → 온프레미스 MySQL
```

**4단계에서 NAT Gateway를 제거하고 Firewall이 아웃바운드를 처리하는 이유**

Firewall은 SNAT(Source Network Address Translation) 기능이 있어서 내부 서버의 아웃바운드 트래픽을 처리할 수 있습니다. NAT Gateway가 UDR 때문에 어차피 동작 안 하니, Firewall이 대신 처리하면 됩니다.

---

### UDR (User Defined Routing)

**비유: 네비게이션 강제 경로 설정**

Azure는 기본적으로 트래픽 경로를 자동으로 잡습니다. UDR은 "무조건 이 경로로 가"라고 강제하는 규칙입니다.

**0.0.0.0/0 → Firewall의 의미**

"목적지가 어디든(0.0.0.0/0) 무조건 Firewall을 먼저 거쳐라"는 뜻입니다. 이렇게 하면 VMSS에서 나가는 모든 트래픽이 Firewall의 검사를 받습니다.

**⚠️ 중요: Propagate Gateway Routes 설정**

UDR과 함께 반드시 설정해야 하는 항목이 있습니다.

```
문제 상황:
VPN Gateway가 온프레미스 경로(10.10.34.0/24)를 Spoke에 자동으로 알려줌
→ Spoke가 "온프레미스는 10.10.34.0/24로 가면 되겠구나" 학습
→ 이 경로가 0.0.0.0/0 UDR보다 더 구체적 (더 구체적인 경로가 우선)
→ 온프레미스 트래픽이 UDR 무시하고 VPN으로 직접 감
→ Firewall 우회됨! ❌

해결책:
Web-Subnet, PE-Subnet 라우트 테이블:
  Propagate Gateway Routes = No
  → 온프레미스 경로를 자동 학습하지 않음
  → 0.0.0.0/0 UDR이 온프레미스 트래픽도 잡아서 Firewall 경유 ✅

GatewaySubnet 라우트 테이블:
  Propagate Gateway Routes = Yes (반드시 활성화)
  → VPN Gateway가 정상 동작하려면 경로 전파가 활성화되어야 함
  → 비활성화하면 VPN 연결 불안정
```

### VPN Gateway

**비유: 본사(Azure)와 서버실(온프레미스)을 연결하는 암호화 전용 전화선**

인터넷을 통해 연결하지만 모든 데이터가 암호화되어 외부에서 내용을 볼 수 없습니다.

**Hub에 배치하는 이유**

```
VPN Gateway를 Hub에 두면:
Hub VPN Gateway ←── 온프레미스
       ↓ Peering (Allow Gateway Transit)
  Spoke VNet
  → Web VM이 Hub의 VPN을 빌려서 온프레미스 DB 접근 가능

VPN Gateway를 Spoke에 두면:
각 Spoke마다 VPN Gateway 필요 → 비용 폭발
Hub-Spoke 구조의 의미 없어짐
```

**Peering Gateway Transit 설정**

```
Hub → Spoke Peering 설정:
  Allow Gateway Transit = true
  → "우리 Hub의 VPN Gateway를 Spoke가 빌려써도 돼"

Spoke → Hub Peering 설정:
  Use Remote Gateways = true
  → "Hub의 VPN Gateway를 빌려서 쓸게"

이 설정 없으면:
Spoke의 Web VM이 VPN 터널을 인식하지 못함
→ 온프레미스 DB(10.10.34.10) 접근 불가
→ WordPress DB 연결 실패
→ 서비스 전체 다운
```

**단일 VPN 구성의 위험성 (SPOF)**

현재 VPN을 1개만 연결합니다. VPN이 끊기면 온프레미스 MySQL 접근이 불가능해져 웹사이트 전체가 다운됩니다.

```
해결 방안 (현재 미적용, Q&A 대비):
1. Active-Active VPN: VPN을 2개 연결. 하나 끊겨도 다른 하나로 연결 (+$140/월)
2. Azure MySQL Replica: VPN 끊기면 Azure DB로 자동 전환 (+$25/월)
3. 온프레미스 WAN 회선 이중화: 물리 회선 2개로 이중화 (온프레미스 비용)
```

### Private DNS Zone

**비유: 사내 전화번호부**

Storage, Redis에 Private Endpoint를 연결하면 IP 대신 도메인 이름으로 접근합니다. DNS Zone이 없으면 "이 도메인 주소가 어디 있는지" 찾지 못해 연결이 실패합니다.

```
필요한 DNS Zone:
- privatelink.blob.core.windows.net  → Storage 접근용
- privatelink.redis.cache.windows.net → Redis 접근용 (3단계~)
```

**Hub VNet에도 링크해야 하는 이유**

App Gateway는 Hub VNet에 있습니다. 백엔드 서버(Spoke에 있는 Storage, Redis)의 도메인을 해석하려면 Hub VNet에서도 DNS Zone을 찾을 수 있어야 합니다. Spoke에만 링크하면 App Gateway가 백엔드 주소를 해석하지 못해 연결이 실패합니다.

**4단계에서 South VNet에도 링크하는 이유**

DR 리전(Korea South)이 활성화됐을 때 South의 App Gateway와 Web 서버가 Storage, Redis에 접근하려면 South Hub/Spoke VNet에도 DNS Zone이 링크되어 있어야 합니다.

### Storage Account

**비유: 잠금장치 있는 창고**

WordPress 미디어 파일, 백업 등을 저장합니다. Private Endpoint로 인터넷에서 직접 접근이 불가능합니다.

**LRS → ZRS → GRS 단계별 전환 이유**

```
1단계 LRS (Locally Redundant Storage):
같은 AZ 안에서 3개 복사
→ 가장 저렴, 최소 내구성

2단계 ZRS (Zone Redundant Storage):
3개 AZ에 각각 1개씩 복사
→ AZ 하나 완전히 불통 되어도 데이터 안전

4단계 GRS (Geo Redundant Storage):
Korea Central에 ZRS + Korea South에 자동 복제
→ 리전 전체 장애 시 South에서 데이터 복구 가능
```

**GRS 주의사항**

GRS는 데이터를 South에 자동으로 복제하지만, 서비스가 자동으로 South Storage를 쓰도록 전환되지는 않습니다. DR 활성화 시 수동으로 Failover 절차를 실행해야 합니다.

### Azure Monitor + Log Analytics

**비유: 건물 전체 CCTV 관제 센터**

모든 서버의 상태(CPU, 메모리, 오류)와 보안 로그를 한 곳에서 실시간으로 봅니다.

**WAF와의 연계**

3단계에서 WAF를 Detection 모드로 시작합니다. Azure Monitor로 WAF 로그를 분석해서 오탐(정상 트래픽을 공격으로 오인한 경우)이 없으면 Prevention 모드로 전환합니다.

## 5. 트래픽 흐름 전체 정리

### 사용자 접속 (1~3단계)
```
사용자 (인터넷)
  ↓ HTTPS
App Gateway + WAF
  ↓
VMSS
  ├→ Redis (세션 확인, 3단계~)
  └→ MySQL (데이터 조회/저장)
```

### 사용자 접속 (4단계)
```
사용자 (인터넷)
  ↓ HTTPS
App Gateway + WAF
  ↓
Azure Firewall
  ↓
VMSS
  ├→ Redis (세션)
  └→ VPN Gateway → Bluemax 방화벽 → MySQL (10.10.34.10:3306)
```

### 서버 아웃바운드 (1~3단계)
```
VMSS
  ↓
NAT Gateway
  ↓
인터넷 (업데이트, 외부 API 등)
```

### 서버 아웃바운드 (4단계, NAT GW 제거)
```
VMSS
  ↓ UDR (0.0.0.0/0 → Firewall)
Azure Firewall (SNAT 처리)
  ↓
인터넷
```

## 6. DR (재해 복구) 구조

**평소**
```
Korea Central (ACTIVE): 정상 운영
Korea South (STANDBY):  VMSS min=0 (꺼둠)
                         Storage GRS 복제본만 있음
```

**Korea Central 전체 장애 발생 시**
```
1. South VMSS 자동 기동 (min=0 → 스케일 아웃, 약 3~5분)
2. DNS TTL 만료 후 South App Gateway IP로 전환 (최대 5분)
3. South VPN Gateway로 온프레미스 MySQL 접근
4. Storage Failover 수동 실행 (자동 전환 아님)
총 전환 시간: 약 5~10분
```

## 8. 자주 나올 질문 (Q&A 대비)

**Q. VPN이 끊기면 어떻게 되나요?**

A. 온프레미스 MySQL 접근 불가 → 웹사이트 전체 다운됩니다. 현재 단일 VPN 구성으로 이 위험을 감수하고 있습니다. 해결책으로는 Active-Active VPN 이중화, Azure MySQL Replica 대기, 온프레미스 회선 이중화 등이 있습니다.

**Q. WAF Detection과 Prevention의 차이는?**

A. Detection은 공격을 탐지하되 통과시키고 로그만 기록합니다. Prevention은 즉시 차단합니다. WordPress 플러그인 오탐 방지를 위해 Detection으로 먼저 운영하다가 로그 확인 후 Prevention으로 전환합니다.

**Q. App Gateway가 Firewall보다 앞에 있는 이유는?**

A. App Gateway v2는 공인 IP와 인터넷 직접 통신이 반드시 필요합니다. Firewall 뒤에 두면 비대칭 라우팅이 발생해 서비스가 마비됩니다.

**Q. DR 리전 활성화는 얼마나 걸리나요?**

A. VMSS 부팅 시간(3~5분) + DNS 전환(최대 5분) = 총 5~10분입니다.

**Q. MySQL을 온프레미스에 두면 속도가 느리지 않나요?**

A. VPN 터널 레이턴시가 추가됩니다. Redis가 세션 캐시를 담당하고 Read Replica가 읽기 부하를 분산해서 실제 DB 직접 접근을 최소화합니다.

**Q. GRS Storage가 자동으로 South에서 동작하나요?**

A. 데이터 복제는 자동이지만 서비스 전환은 수동 Failover 절차가 필요합니다.

**Q. Propagate Gateway Routes를 왜 Spoke는 No, GatewaySubnet은 Yes로 설정하나요?**

A. Spoke를 No로 설정하면 VPN이 온프레미스 경로를 자동으로 알려주지 않아서 모든 트래픽이 0.0.0.0/0 UDR에 걸려 Firewall을 통과합니다. GatewaySubnet은 Yes로 해야 VPN Gateway가 정상적으로 경로를 관리하고 온프레미스와 통신할 수 있습니다.
